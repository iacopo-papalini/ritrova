# FaceRecog UX Audit

Date: 2026-04-10

Scope: code-based UX assessment of the current app, with `docs/bugs.md` treated as the source of already-known issues and planned features.

Method: reviewed the main user journeys across the FastAPI routes, Jinja templates, and client-side interaction patterns. This is a product/interaction audit, not a security review.

## Executive Summary

The app already has a strong core workflow for a personal archive tool:

- unnamed clusters are visually fast to review
- singletons, compare, merge suggestions, and find-similar provide real correction tools rather than a single flat naming screen
- the dark gallery redesign makes the product feel intentional instead of admin-like
- the lightbox, month grouping, and species split are meaningful usability improvements

The main UX weakness is not lack of capability, but fragmentation. The product has several powerful flows, but they are not yet tied together into a consistently guided review experience. Users can do most important tasks, but they often need to infer the next best step, recover context after actions, and tolerate full-page reloads or route inconsistencies.

In practical terms:

- cluster triage is the strongest journey
- correction and auditing flows exist, but feel advanced and under-explained
- retrieval/search is still shallow compared to the value of the data model
- cross-page consistency is the biggest source of friction
- quality-of-life features like undo, progress indicators, and saved context would have outsized impact

## Product Model

The app is effectively serving five user goals:

1. Ingest a photo library and produce workable face/pet candidates.
2. Turn unnamed clusters into named people or pets quickly.
3. Clean up bad detections and wrong assignments.
4. Retrieve photos of a person or pet over time.
5. Audit the dataset for remaining mistakes.

That is the right model. The app is not just a face tagger; it is a review environment for a noisy archive. The best UX direction is therefore not “more screens”, but “stronger review loops”.

## What Already Works Well

### 1. The app understands the real workflow

The route set is good: clusters, singletons, compare, merge suggestions, person detail, photo detail, and find-similar are all relevant tools for iterative identity cleanup.

This matters because face recognition data is never clean enough for a single naming pass. The app reflects that reality.

### 2. The visual system fits the task

The current dark gallery aesthetic is a good fit for photo review work:

- high-density thumbnail grids are readable
- warm accent color gives action affordance without looking like an ops dashboard
- month grouping on person detail matches how people remember archives
- the lightbox adds important contextual confidence without leaving the current task

### 3. There is good support for ambiguity

Several flows acknowledge that identity work is probabilistic:

- singleton hints
- merge suggestions
- compare view
- find-similar by threshold
- ranked assignment candidates in cluster detail

That is the right product instinct. The app is strongest when it behaves like an assistant rather than a rigid classifier.

### 4. The app supports both batch and surgical cleanup

Users can:

- assign an entire cluster
- exclude a single face from a cluster
- dismiss a cluster
- bulk-dismiss or bulk-assign selected singletons
- move likely misassigned faces between identities
- unassign one wrong face from a named person

This is a strong foundation.

## Known Issues Already Tracked In `docs/bugs.md`

These are already on record and should remain part of the UX roadmap:

- `FEAT-5`: global single-step undo
- `FEAT-6`: find photos containing multiple people/pets
- `BUG-5`: verify scanner exclusion behavior for generated files

Two of these are especially important UX items:

- Undo is not a minor enhancement. In a curation tool, it changes user confidence across every write action.
- Multi-person photo retrieval is a high-value retrieval feature because it turns the dataset into something users can actually browse for memories, not just identities.

## Deep UX Findings By Journey

## 1. Entry And Orientation

### Strengths

- the dashboard is visually clear
- key counts are prominent
- there is a direct path into unnamed clusters

### Friction

- the dashboard mostly acts as a stats page, not a control center
- it does not clearly answer “what should I do next?”
- it does not help users distinguish backlog types:
  - unnamed clusters
  - unclustered singletons
  - likely bad detections
  - likely merges
  - likely misassignments

### UX opportunity

Turn the dashboard into a review inbox.

High-impact additions:

- “Next best action” cards
- separate counts for people and pets, or a top-level scope switch
- “review queue” summaries such as:
  - `23 unnamed clusters`
  - `114 unmatched singles`
  - `8 likely merges`
  - `12 suspected misassignments`
- one-click resume links into the exact review queue

The current dashboard tells the user what exists. A stronger dashboard would tell them where to spend the next five minutes.

## 2. Global Navigation And Information Architecture

### Strengths

- the top nav is visually clean
- the humans/pets split is conceptually simple
- the main work areas are exposed directly

### Friction

- the app is in a partial transition between old and new URL schemes
- some screens use `/people/...` and `/pets/...`
- some templates still link to legacy `/persons/...`, `/clusters/...`, `/compare`, or `/search`
- some forms submit to legacy endpoints and rely on redirects
- search is effectively a separate mode rather than a first-class part of the same people/pets navigation model

This does not just create technical inconsistency. It creates cognitive inconsistency:

- users can lose current scope
- page relationships feel less predictable
- some actions feel like “jumping out” of the current mode

### UX opportunity

Pick one navigation grammar and apply it everywhere:

- one scope model
- one naming model
- one route pattern
- one back-navigation pattern

Product-wise, “People” and “Pets” should feel like two datasets inside the same tool, not semi-separate apps with shared plumbing.

## 3. Unnamed Cluster Review

### Strengths

This is the best journey in the app today.

- cluster cards are dense and scannable
- hover hint is smart and low-noise
- cluster detail provides ranked suggestions, create-new, dismissal, selection, and per-face exclusion
- users can correct the cluster before assigning it

This is a solid review surface.

### Friction

- there is no explicit “skip and continue” action
- there is no visible notion of progress through the backlog
- after actions, the app often reloads or redirects without preserving a sense of sequence
- per-face removal is good, but there is no “keep only selected” or “split out selected as new cluster/person” workflow
- the suggested candidate list shows count and similarity, but not enough context to build trust quickly

### UX opportunity

Improve cluster detail as a review station:

- add `Skip`, `Previous`, and `Next review item`
- show queue position, for example `Cluster 12 of 87`
- allow “assign selected faces only” or “spin selected into new identity”
- provide richer suggestion cards:
  - sample thumbnails from the target person/pet
  - last-seen date range
  - number of photos
- add lightweight result messaging after writes instead of silent redirects

This screen is already strong enough to become the product’s main cockpit.

## 4. Singleton Review

### Strengths

- singleton hints are valuable
- multi-select batch assignment is efficient
- dismiss-as-not-a-face is available

### Friction

- “singletons” are conceptually advanced; the UI does not explain when the user should work here versus in clusters
- hints are only available as a hover-layer interaction
- there is no explicit “create new person/pet from selected singletons” flow
- there is no side-by-side evidence for why a hint is suggested
- assignment uses a long dropdown, which will degrade badly as the library grows

### UX opportunity

Make singletons feel like a guided catch-up queue:

- explain the purpose in user language:
  - “faces that did not form a group”
  - “often one-off photos, blurry detections, or missed identities”
- let users promote selected singletons into a new name directly
- replace or supplement the dropdown with search-based assignment
- add a “show top 3 candidate matches” interaction
- add optional grouping by month or source album to provide context

This is a high-value screen but currently feels more like an internal tool than a polished user workflow.

## 5. Person/Pet Directory

### Strengths

- simple and visually clean
- works for browsing a moderate number of identities

### Friction

- the page is basically a flat card list
- there is no sorting control
- there is no filtering by face count, recency, or likely uncertainty
- there is no “recently updated” or “needs review” concept
- the label “Names” is concise, but also vague; it undersells that this is the main identity directory

### UX opportunity

Turn the directory into a working catalog:

- add sort options:
  - name
  - most faces
  - recently changed
  - recently seen
- add filters:
  - low sample count
  - no recent review
  - likely mixed identity
- consider renaming the nav item from `Names` to `People` or `Directory`

This page should help users find the identities that need attention, not just list them.

## 6. Person/Pet Detail

### Strengths

- month grouping is excellent for archival memory
- splitting face samples and photos into tabs fixes the original overload problem
- the per-face unassign action is essential and now present
- this is one of the few screens that feels tied to how users remember real people over time

### Friction

- there is no summary insight beyond counts
- there is no confidence/review status for the identity
- the “face samples” tab is useful for correction, but not for interpretation
- the photos tab is a plain grid rather than a discovery surface
- no quick way exists to find co-occurring people/pets from this screen
- no “recently added faces” or “possible wrong faces” subsection exists
- rename and delete are available, but merge-with-another-person is not surfaced here even though the backend supports merging

### UX opportunity

Make detail pages feel like identity hubs:

- add a small summary panel:
  - first seen / last seen
  - number of photos
  - number of face samples
  - source types: photo vs video frames
- add “possible mistakes” shortcuts:
  - compare with another identity
  - review find-similar
  - recent unassigned faces
- expose merge-person action directly on the page
- on the photos tab, support:
  - larger “hero” recent photos
  - timeline scrubbing
  - filter to photos with multiple faces

The person page should be the canonical place where users both admire and audit an identity.

## 7. Photo Detail

### Strengths

- seeing boxes on the original photo is important and well chosen
- the metadata block is useful
- face cards with assign/unassign actions give a good correction loop

### Friction

- `history.back()` is fragile as the only explicit navigation model
- there is no next/previous photo browsing flow
- there is no quick “open containing person” set for all known identities in the photo
- there is no bulk action for all unknown faces in a photo
- the page does not emphasize group-photo workflows, even though it is well positioned for them

### UX opportunity

This page could become the best place for contextual correction:

- add stable back links when the user arrives from a person page
- add next/previous within the current result set
- add “assign all selected unknown faces”
- add “show everyone in this photo”
- add one-click “find more photos with these same people/pets” once `FEAT-6` exists

Right now the page is useful, but isolated.

## 8. Search And Retrieval

### Strengths

- simple name search exists
- low learning curve

### Friction

This is one of the weakest areas relative to the app’s data value.

- search appears to be name-only
- it is not integrated into people/pets scope consistently
- it does not search by:
  - aliases
  - face count
  - time
  - co-occurrence
  - album/path
- results are only identity cards, not photos

For a memory archive product, users eventually want retrieval more than labeling.

### UX opportunity

Search should become a retrieval layer, not just a directory filter.

Recommended roadmap:

1. Fix people/pets scope consistency.
2. Add fast typeahead assignment/search components.
3. Add photo search by selected identities.
4. Add “photos containing all of these”.
5. Add path/date narrowing.

`FEAT-6` is therefore not just a bonus feature. It is the beginning of the app’s second major use case after labeling.

## 9. Merge Suggestions

### Strengths

- the feature is valuable
- similarity threshold pills make sense
- the split between named-vs-unnamed and cluster-vs-cluster is understandable

### Friction

- the page is transactional rather than investigative
- there is not enough supporting context to make merge confidence feel high
- there is no “not now” or “hide this suggestion” action
- there is no visible reason why the suggestion appeared beyond a raw percentage
- for named identities, users likely need more evidence before accepting a merge-like action

### UX opportunity

Treat merge suggestions as decisions that require trust-building:

- show representative date spread
- show more photo diversity in samples
- link directly into compare or person detail
- allow dismissing a suggestion temporarily
- explain whether the pair is:
  - named + unnamed
  - unnamed + unnamed
  - near-duplicate cluster candidates

The raw similarity number is useful but not sufficient as a user-facing explanation.

## 10. Compare

### Strengths

- this is a genuinely powerful correction tool
- the concept is clear once understood
- the similarity deltas are useful

### Friction

- this is an expert feature with almost no onboarding
- the screen assumes the user already knows which two identities to compare
- there is no help in picking likely-confusable pairs
- there is no photo context or lightbox access in the face grid
- actions rely on full reload rather than progressive review

### UX opportunity

Elevate compare from “manual diagnostic tool” to “guided audit tool”:

- add suggested compare pairs:
  - recently merged candidates
  - same-era lookalikes
  - identities with suspicious cross-similarity
- add a quick lightbox/photo context on each face
- support “move and continue” without page reset
- explain the model in plain language:
  - “These faces currently belong to A but look more like B.”

Compare is strategically important because it is where users repair trust in the system.

## 11. Error Prevention, Confidence, And Recovery

### Strengths

- confirmation prompts exist for destructive actions
- there are specific correction tools, not just delete/recreate

### Friction

- most write actions still feel high-risk because recovery is weak
- many actions end in `location.reload()` or silent DOM removal
- there is little explicit feedback that the intended change succeeded
- users can lose context, selection state, or momentum after an action
- there is no durable “last action” visibility

### UX opportunity

`FEAT-5` should be treated as a core UX feature, not a backend convenience.

Beyond undo, add:

- lightweight success toasts
- action summaries, for example:
  - `Assigned 14 faces to Teresa`
  - `Removed 1 face from Figaro`
- keep users in place after routine corrections
- preserve selection and scroll where possible

The app currently gives users tools. The next step is to make those tools feel safe.

## 12. Micro-Interaction Quality

### What is good

- hover affordances are generally tasteful
- selection overlays are readable
- infinite scroll is appropriate for dense grids

### What still feels rough

- heavy reliance on browser `confirm()` dialogs
- many actions require reloads even when they could be local updates
- hover-only affordances can be harder on touch devices
- no empty-state illustrations or richer guidance in advanced screens
- no keyboard shortcuts for high-volume review work

### UX opportunity

Small improvements would compound significantly:

- custom confirmation modal with clearer copy
- keyboard shortcuts:
  - next item
  - assign top candidate
  - dismiss
  - open lightbox
  - undo
- toast feedback instead of reload-only confirmation
- touch-friendly secondary actions for mobile/tablet review

This is the kind of product where micro-efficiency matters because users may repeat actions hundreds of times.

## New UX Issues Not Clearly Tracked In `docs/bugs.md`

These are not just code inconsistencies; they affect user flow directly.

### 1. People/Pets scope is still inconsistent in several screens

Observed patterns include:

- global search submits to `/search`
- search results link to legacy `/persons/...`
- compare form submits to `/compare`
- some merge suggestion links point to legacy `/persons/...`
- cluster cards link to legacy `/clusters/{id}`

Impact:

- users can lose scope unexpectedly
- pet workflows can route back through people-oriented entry points
- the app feels less coherent than it really is

Recommendation:

- unify the route model and link model everywhere
- treat mixed routing as a product bug, not just a refactor cleanup

### 2. Search is underpowered relative to the rest of the app

The rest of the app implies a rich archive browser. Search currently behaves more like a simple directory filter.

Recommendation:

- prioritize retrieval features after undo
- expand search from names to photos, co-occurrence, and timeline-aware browsing

### 3. Several advanced flows are discoverable only if the user already understands the system

This is especially true for:

- singletons
- compare
- find-similar
- merge suggestions

Recommendation:

- add short “what this page is for” copy
- add suggested entry points from dashboard and person pages

### 4. The app lacks a coherent backlog/progress model

Users can work, but they are not clearly told:

- what remains
- what changed
- what is most valuable next

Recommendation:

- introduce review queue concepts and progress indicators

## Prioritized UX Roadmap

## Tier 1: Highest Impact

- Implement global undo and action toasts.
- Unify people/pets routing, scope preservation, and cross-page links.
- Turn the dashboard into a “next best actions” review inbox.
- Improve search and retrieval direction, starting with scope consistency and `FEAT-6`.

## Tier 2: Review Loop Quality

- Add queue progress, skip, and continue actions in cluster review.
- Improve singleton assignment with search-based assignment and “create new from selected”.
- Surface merge-person and review shortcuts directly on person detail.
- Improve compare with suggested candidate pairs and photo context.

## Tier 3: Workflow Polish

- Replace browser confirms with app-native dialogs.
- Reduce full reloads after routine operations.
- Add keyboard shortcuts for high-volume curation.
- Improve empty states and explanatory copy on advanced pages.

## Product Direction Recommendation

The best version of this app is not just “a face naming UI”. It is a personal archive review environment with two complementary modes:

- curation mode: name, merge, dismiss, correct
- retrieval mode: find photos and relationships across time

The codebase is already much closer to that vision than the current UX presentation suggests. The main opportunity is to make the product feel like one coherent review system rather than a set of strong individual tools.

## Questions Worth Clarifying Later

I could complete a sharper second-pass audit with answers to these product questions:

- Is the primary success metric faster labeling, better accuracy, or better photo retrieval after labeling?
- Should pets and people eventually support combined retrieval workflows by default?
- Do you want the app optimized mainly for desktop curation sessions, or also for casual tablet/phone browsing?
- Is “compare” intended for power users only, or should it become part of the mainstream cleanup flow?
